# Roadmap: Janus 2.0 — Frontend Consolidation

## Overview

This milestone closes out the ~80% finished frontend consolidation. It starts by landing the canonical role-aware route guard so that every route migrated afterward adopts it, then completes the in-progress route component-split, removes duplicate non-admin/admin route pairs, regenerates the route tree, and finally repairs the broken test suites so the consolidated app has working regression protection. No new product features and no backend security work — those are parked for the next milestone.

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): Planned milestone work
- Decimal phases (2.1, 2.2): Urgent insertions (marked with INSERTED)

Decimal phases appear between their surrounding integers in numeric order.

- [ ] **Phase 1: Canonical Guard** - Collapse to one role-aware `ProtectedRoute` and migrate all routes onto it
- [ ] **Phase 2: Route Split & Tree** - Finish the component-split, remove duplicate routes, regenerate the route tree
- [ ] **Phase 3: Test Repair** - Repair E2E specs and get the Vitest unit run green (Playwright excluded)
- [ ] **Phase 4: Cleanup** - Delete the stale migration script

## Phase Details

### Phase 1: Canonical Guard
**Goal**: One canonical role-aware guard exists and every route file uses it, with no role-based access regression.
**Depends on**: Nothing (first phase)
**Requirements**: GUARD-01, GUARD-02, GUARD-03
**Success Criteria** (what must be TRUE):
  1. `ProtectedRoute.tsx` (PascalCase, `allowedRoles`-aware) is the only guard component in the codebase
  2. No file imports the legacy `components/protected-route.tsx`, and the file no longer exists on disk
  3. All ~15 migrated route files declare an `allowedRoles` appropriate to their audience (admin routes admin-only, etc.)
  4. `npm run build` passes after migration
  5. Manually visiting an admin route as a non-admin role still redirects (role guard not regressed)
**Plans**: TBD
**UI hint**: yes

### Phase 2: Route Split & Tree
**Goal**: The route layer is consolidated — thin lazy-loading wrappers, no duplicate route pairs, and a freshly regenerated route tree with all new directories committed.
**Depends on**: Phase 1
**Requirements**: ROUTE-01, ROUTE-02, ROUTE-03, ROUTE-04, ROUTE-05
**Success Criteria** (what must be TRUE):
  1. `admin/dashboard`, `admin/person`, and `admin/discussions` are thin route wrappers that lazy-load their `_component.tsx` implementations
  2. The new route directories (`admin/dashboard/`, `admin/discussions/`, `admin/person/_component.tsx`) are committed to git, not untracked
  3. Duplicate non-admin/admin route file pairs are removed or unified into a single role-parameterized route (no double-maintenance copies remain)
  4. `routeTree.gen.ts` is regenerated by the codegen (not hand-edited) and reflects the current route set
  5. `npm run build` passes with the regenerated tree
