## 1. Foundation — Role-Gated ABAC Engine

**What it does:** A shared in-memory "world" of 6 canonical units (2 military, intel, infra, industry, home guard), each with subjects, resources, and an attribute store. A pure-computed ABAC engine evaluates any subject/resource/domain triple live and shows a per-rule trace.

**How to test it:**

| Step | What to do | What to observe |
|------|-----------|-----------------|
| 1 | Open the demo shell | Non-dismissable `[DEMO / MOCK]` banner at the top — cannot be hidden |
| 2 | Open the **Decision Explorer** tab | Unit and subject pickers, resource picker, domain selector |
| 3 | Pick any subject + resource + domain, press Evaluate | ALLOW or DENY with a per-rule trace (clearance, tier, domain check) |
| 4 | Switch the resource to one with a security hold | DENY via "deny override" appears separately from base rule failures |
| 5 | Change domain (computer → data → physical) | Each domain evaluates independently; a tier failure in one domain doesn't affect others |
| 6 | Use the **Role Switcher** (top right) | Switch to Manager — grant/revoke buttons disappear; switch to Approver — they reappear |
| 7 | As Approver, grant an attribute to a subject | Event appears in the event log; the live decision updates immediately |

**Key questions for prioritization:**
- Is the per-rule trace legible to a non-developer? (Gate: DEMO-03)
- Are the 8 roles realistic enough for the actual operating context?
- Should "clearance" ever come from a real external feed, or always mocked?

## 2. Federation Hub — Discovery Without Disclosure

**What it does:** A hub shows which entities hold authorization information about a subject — pointer only, no details. A typed inter-entity message transcript traces the full exchange (publish → discover → request → response). A requester's credential is signed and verified before the holder evaluates release.

**How to test it:**

| Step | What to do | What to observe |
|------|-----------|-----------------|
| 1 | Open the **Federation Hub** tab | Three-panel layout: Hub Discovery, Exchange Transcript, Credential Verify |
| 2 | **Hub Discovery panel** — query for a subject | Pointer list appears: "Unit X holds info in domain Y" — no details shown; explicit "what the hub does NOT store" callout |
| 3 | **Credential Verify panel** — inspect auto-verification on load | Side-by-side: rogue credential (forged issuer) → REJECTED `[MOCK]`; valid credential → VERIFIED |
| 4 | **Exchange Transcript panel** — step through the 4-stage exchange | Publish, Discover, Request (credential attached), Response — each step labelled; verdict shown at end |
| 5 | Note that "holder-gated": the response is DENIED unless the holder's ABAC policy authorizes release | Try with a subject who lacks the required attribute — see DENY with trace |

**Key questions for prioritization:**
- Is the hub discovery model what we actually want? (Pointer-only, vs. a richer index)
- Should credentials be HMAC-signed (as mocked) or move to asymmetric keys / real VC format?
- Is the 4-step typed-contract transcript the right abstraction for the real network protocol?

## 3. Entity Console — Per-Unit Holdings

**What it does:** Per-unit view showing each unit's inbox (incoming requests), outbox (sent requests), and held attribute data. Lets you inspect the federation flow from the holder's perspective.

**How to test it:**

| Step | What to do | What to observe |

| 2 | Select each of the 6 units | Holdings differ by unit; inbox/outbox populated from the federation exchange |

**Key questions for prioritization:**

## 4. Audit — Append-Only Log & Point-in-Time Reconstruction

**What it does:** Every action is recorded in an append-only event log. Current access state is a materialized O(1) projection over the log. A timeline slider lets you reconstruct "who can access resource R as of time T" — revocations and holds are reflected correctly at T′ without altering prior entries.

**How to test it:**

| Step | What to do | What to observe |
|------|-----------|-----------------|
| 1 | Open the **Audit** tab | Event log on the left; "Who Can Access" panel on the right; timeline slider at top |
| 2 | Perform some actions in Decision Explorer (grant, revoke, hold) | Events appear in the log in real-time |
| 3 | Move the timeline slider backward | "Who Can Access" panel updates to reflect access state at that point in time |
| 4 | Grant access, then revoke it; scrub slider across that boundary | Access shows ALLOW before revocation, DENY after — prior entries unchanged |
| 5 | Apply a Security Officer hold; check slider | Hold appears as a distinct event; access DENY shown with "override" marker |

**Key questions for prioritization:**
- Is point-in-time reconstruction the right UI pattern for real operators, or is a diff/change view more useful?
- Should the audit log be the primary persistence model in the real build, or a secondary record alongside a mutable state store?
- Is "who can access resource R at time T" the right query, or should it be "what changed for subject S between T1 and T2"?

## 5. Context & Policy — Divergence, Obligations, Shielding

**What it does:** Three mechanisms on one view:
1. **Policy divergence** — the same request sent to two different entities resolves differently based on each entity's own release policy.
2. **Deployment obligations** — toggling a subunit's status from HOME to ABROAD activates a support-obligation grant to obligated units; returning to HOME turns it off (no stored grant).
3. **Directional shielding** — a non-allowlisted entity is DENIED access to shielded intel/industry data even with standing access; an allowlisted entity succeeds.

**How to test it:**

| Step | What to do | What to observe |
|------|-----------|-----------------|
| 1 | Open the **Context / Policy** tab | Three panels: Policy Divergence, Deployment Obligations, Directional Shielding |
| 2 | **Policy Divergence** — send the same request via Entity A and Entity B | Different ALLOW/DENY outcomes; each shows its own policy trace |
| 3 | **Deployment Obligations** — toggle a subunit from HOME → ABROAD | Support-obligation access turns ON (shown in the obligation panel) |
| 4 | Toggle ABROAD → HOME | Support-obligation access turns OFF — no stored grant was created; it's computed from the deployment attribute |
| 5 | **Directional Shielding** — attempt access to shielded data from a non-allowlisted entity | DENY, with "not on allowlist" as the trace; no other rule matters |
| 6 | Same request from an allowlisted entity | ALLOW (if other rules pass) |

**Key questions for prioritization:**
- Is deployment-status-as-attribute realistic? How does the system learn a unit is deployed?
- Is directional shielding (allowlist) the right model for industry data, or should it be a tag/label on the resource?
- For per-entity policy divergence: how is a policy actually authored in the real system? (UI needed?)
