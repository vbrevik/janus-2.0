<objective>
Close the backend's live server-side security holes in the same pass as the digital-resource work (SEC-01..04): give the unauthenticated `vendor_relations` HTTP handlers `AuthGuard` (SEC-01), gate every write/mutation handler and the audit read with `role_has_permission` so authentication is no longer mistaken for authorization (SEC-02), remove the hardcoded JWT-secret fallback so startup fails loud (SEC-03), and restrict CORS from wildcard to the single dev origin while keeping credentials (SEC-04).

Purpose: SEC-01..04 are first-class requirements, not polish — they share the backend trust boundary with the new issue endpoints and touch `rocket_setup.rs`, which this phase already modified for the migration repair and the domain mount. Reuse existing primitives only: `AuthGuard` and `role_has_permission`. No new auth framework.
Output: hardened handlers + rocket_setup, a permission-seed migration, JWT_SECRET wired into compose, and security integration tests.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → any backend handler | Unauthenticated/unauthorized access to mutations and sensitive reads |
| environment → JWT verification | A missing/known secret lets anyone forge admin tokens |
| browser origin → CORS | A wildcard-with-credentials origin enables cross-site credentialed requests |

| T-11-13 | Spoofing / missing-auth | vendor_relations unauthenticated endpoints (SEC-01) | mitigate | AuthGuard added to all 4 vendor_relations HTTP handlers; unauthenticated request rejected (test) |
| T-11-14 | Elevation of Privilege / broken access control (SEC-02) | any authenticated actor writes to any domain | mitigate | role_has_permission gate on every write/mutation + audit read; 403 (not 500) when role lacks the key; keys seeded |

| T-11-16 | Information Disclosure / cross-origin (SEC-04) | AllowedOrigins::all() + allow_credentials(true) | mitigate | CORS restricted to http://localhost:15510, credentials retained; foreign origin not granted (test) |
