<objective>
Build the React Query data layer the loader and the two issuing forms consume: a `useDigitalResourcesWorld` query hook, `useIssueGrant`/`useIssueDelegate` mutation hooks that dispatch their server response straight into `WorldState`, and three small pure helpers (`hasStoredToken`, `getStoredUserRole`, `classifyLoaderState`) that turn raw auth/query state into the exact loader-state and role-gate decisions 12-UI-SPEC.md pins.

Purpose: every later UI plan (12-04, 12-05, 12-06) needs a single, tested place that knows how to fetch+map the world, issue a grant/delegate and reflect it back into state, and decide "is there a token / is the caller admin / which of the six loader states are we in" — duplicating any of this logic per-component would risk the three states drifting out of sync with each other.
Output: `frontend/src/demo/hooks/use-digital-resources.ts` (+ test file covering the three pure helpers).
</objective>

  <behavior>
    - `hasStoredToken(): boolean` — `!!localStorage.getItem("token")`. Pure, no React dependency.
    - `getStoredUserRole(): string | null` — reads `localStorage.getItem("user")`; if absent, returns `null`; if present but `JSON.parse` throws or the parsed value has no string `.role`, returns `null` (never throws).
    - `type LoaderState = "missing-token" | "loading" | "unauthorized" | "error" | "empty" | "success"`.
    - `classifyLoaderState(hasToken: boolean, snapshot: { isLoading: boolean; isError: boolean; error: unknown; isSuccess: boolean; data: DigitalResourceWorld | undefined }): LoaderState` — order of checks: (1) `!hasToken` → `"missing-token"`; (2) `snapshot.isLoading` → `"loading"`; (3) `snapshot.isError` → `"unauthorized"` if `snapshot.error instanceof ApiError && snapshot.error.status === 401`, else `"error"`; (4) `snapshot.isSuccess && snapshot.data` → `"empty"` if `data.networks.length === 0 && data.platforms.length === 0 && data.applications.length === 0`, else `"success"`; (5) fallback `"loading"` (the brief window before the query has started when `enabled` just flipped true).
    - `useDigitalResourcesWorld(hasToken: boolean)` — `useQuery({ queryKey: ["digital-resources", "world"], queryFn: async () => mapWorldResponse((await apiFetch<ApiResponse<DigitalResourceWorldResponse>>("/api/digital-resources/world")).data), enabled: hasToken, retry: false, refetchOnWindowFocus: false })`. The returned `UseQueryResult` object is structurally compatible with `classifyLoaderState`'s `snapshot` parameter (same field names) — callers pass it straight through, no adapter needed.
    - `useIssueGrant()` — `useMutation` whose `mutationFn` POSTs `{ resource_id, person_id, actor_org_id, valid_from, valid_until }` (all strings, `valid_from`/`valid_until` as ISO date strings or `null`) to `/api/digital-resources/grants`, unwraps the `ApiResponse<RawResourceAccessGrant>` envelope, and maps the raw grant to `ResourceAccessGrant` via `parseNullableDate` (reused from 12-02's mapper) for `valid_from`/`valid_until`. Internally calls `useWorldDispatch()` and, in `onSuccess`, dispatches `{ type: "UPSERT_RESOURCE_GRANT", grant: <mapped grant> }`.
    - `useIssueDelegate()` — mirrors `useIssueGrant()` exactly: POSTs `{ resource_id, delegate_type, delegate_person_id, delegate_org_id, granted_by_org_id, valid_from, valid_until }` to `/api/digital-resources/delegates`, maps the response, dispatches `UPSERT_RESOURCE_DELEGATE` on success.
