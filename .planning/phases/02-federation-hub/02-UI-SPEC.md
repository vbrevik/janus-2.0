# Phase 2 — UI Design Contract

> Visual and interaction contract for the Federation Hub phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope:** a new **Federation surface** added to the existing demo island (`frontend/src/demo/`, served by `demo.html`), reachable via a **throwaway interim view switch** alongside the existing Decision Explorer (D2-04). The surface is a single low-density scrolling page, top→bottom (D2-03): **Hub Discovery** (FED-01) → **Exchange Transcript** (FED-02) → **Credential Verify** (FED-03, side-by-side) → **Unit Console** (FED-04 inbox/outbox, one unit via dropdown). The persistent `[DEMO / MOCK]` banner + role switcher header stay outside the swappable region (R7).
>
> **This is throwaway-interim, demo-island, frontend-only UI.** Phase 4 owns the coherent shell + navigation/legibility polish — do NOT over-invest here. This contract **inherits Phase 1's design system verbatim** (`01-foundation/01-UI-SPEC.md`); it documents how the Federation surface *reuses* that vocabulary, not a fresh design system. Every new component reuses the proven spike/demo visual primitives (`demo/components/ui.tsx` — `Card`, `Field`, `Select`, `Pill`, `MockTag`, `DecisionTrace`). Nothing here is reinvented.

## Spacing Scale

Declared values (multiples of 4). **Inherited verbatim from Phase 1** — maps directly to the Tailwind utilities the demo already uses (`gap-4`, `p-4`, `space-y-4`, `grid-cols-2 gap-4`, `px-2 py-0.5`, `px-3 py-1.5/py-2`). The Federation surface reuses this rhythm; it adds no new spacing tokens.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Pill/`MockTag` inline padding, glyph-to-text gaps in envelope/rule rows (`gap-1`) |
| sm | 8px | Compact element spacing (`gap-2`), stage-trigger button padding, transcript row gaps (`space-y-0.5`/`space-y-2`) |
| md | 16px | Default panel + card spacing (`p-4`, `gap-4`, `space-y-4`) — the dominant rhythm of every Federation panel and the side-by-side credential grid (`grid-cols-2 gap-4`) |
| lg | 24px | Section padding between the four stacked panels (Hub Discovery / Transcript / Credential Verify / Unit Console) |
| xl | 32px | Outer demo container padding (`max-w-5xl px-6`) |
| 2xl | 48px | Major break between header chrome + interim toggle row and the view body |
| 3xl | 64px | Page-level top/bottom breathing room on wide viewports |

Exceptions (carried verbatim from the proven spike/demo components — do NOT "tidy" to the 8-pt grid; these are the SAME three documented in Phase 1 and are the ONLY permitted off-grid values):
- Pill/`MockTag` vertical padding `py-0.5` = 2px (`demo/components/ui.tsx:23,35`).
- Stage-trigger / action-button vertical padding `py-1.5` = 6px; primary "send"-style buttons use `py-2` = 8px (on-grid).
- `DecisionTrace` rule-row vertical gap `space-y-1.5` = 6px (`demo/components/ui.tsx:108`); transcript line list uses `space-y-0.5` = 2px (`Spike005Contract.tsx:145`) to keep the four envelopes tight and scannable.
- Guardrail: any NEW spacing the executor adds beyond these documented sub-grid values MUST come from the 4/8/16/24/32/48/64 token set.

## Typography

**Inherited verbatim from Phase 1.** Three roles + display, mapped to Tailwind `text-xs/sm/lg/xl`. Carrying weights: 400 body / 500 medium-emphasis (`font-medium`) / 600 semibold-heading (`font-semibold`). Exception: 700 bold (`font-bold`) on the single ALLOW/DENY / TRUSTED/REJECTED verdict line only — no other element uses 700.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label (uppercase, tracked field/section/panel labels, `Card` titles) | 12px (`text-xs`) | 600 (`font-semibold`), `uppercase tracking-wide` | 1.4 |
| Body / control text / helper prose (panel intros, "hub does NOT store" callout, deny reasons) | 14px (`text-sm`) | 400 regular; 500 (`font-medium`) for emphasis (unit names, rule names) | 1.5 |
| Heading (panel section heads where larger than a `Card` title) | 18px (`text-lg`) | 600 (`font-semibold`) | 1.3 |
| Display (ALLOW/DENY verdict in `DecisionTrace`; `✓ TRUSTED` / `✗ REJECTED` credential verdict) | 20px (`text-xl`) | 700 (`font-bold`) | 1.2 |

Notes:
- The **typed envelope transcript lines** and the credential **signature preview** use `font-mono text-xs` (12px monospace), matching Phase 1's audit-log treatment (`DecisionExplorer.tsx:304`, `Spike006Trust.tsx:107`). This is the federation-specific mono usage — alignment of `PUBLISH/DISCOVER/REQUEST/RESPONSE → unit` lines.
- The credential-outcome card verdict (`✓ TRUSTED` / `✗ REJECTED`) carries `font-bold` per the proven spike (`Spike006Trust.tsx:133`) — it is part of the single permitted 700-weight verdict class, NOT a new weight.
- Do NOT add a 5th size or a 3rd carrying weight; the `[MOCK]`/`[DEMO]` tags, pills, and helper captions all live at `text-xs`.

## Color

**Inherited verbatim from Phase 1.** The base palette is the existing shadcn slate theme; the demo island paints with plain-Tailwind semantic classes (slate/green/red/blue/amber). **Semantic decision colors (green/red/amber/blue) are load-bearing signal, not decorative accent** — they encode federation outcomes (verify pass/fail, ALLOW/DENY release) and MUST be reserved for those meanings only. The amber `[MOCK]` treatment is the security control (MODEL-03) and stays visually distinct from the green/red decision palette.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | white / `bg-slate-50` | Page + Federation view-body surface; the "what the hub does NOT store" callout panel and empty-state panels (`bg-slate-50`) |
| Secondary (30%) | `Card` white + `border-slate-200`; `text-slate-400/500/600/700` | Cards/panels (Hub Discovery, Transcript, Unit Console), header chrome, the interim toggle row, helper/muted prose, transcript line text (`text-slate-600`) |
| Accent (10%) | blue (`bg-blue-100 text-blue-800` pill / `border-blue-300 bg-blue-100` selected control) | Reserved list below |
| Destructive | red (`red-700` solid / `red-50` + `red-200` surface) | DENY release, deny overrides, and the REJECTED credential outcome only (see reserved list) |

Accent reserved for (explicit — never "all interactive elements"):
- The **blue accent** is reserved for: the holding-unit / clearance / info `Pill` (`tone="blue"`, e.g. `<Pill tone="blue">{unitName(p.holdingUnit)}</Pill>` in Hub Discovery + released-record clearance), the currently-selected control state, and focus rings. It is NOT used to paint stage-trigger buttons or the toggle.

Federation semantic palette (fixed meanings — the executor must NOT repurpose these hues; they extend the Phase 1 contract to the new surface):
- **Green** (`green-700` solid / `green-50` + `green-200` surface): ALLOW release verdict, passing rule rows (`✓`) in the inbox `DecisionTrace`, the `✓ TRUSTED` valid-credential outcome card, and the RELEASED state in the transcript/outbox.
- **Red** (`red-700` solid / `red-50` + `red-200` surface): DENY release verdict, failing rule rows (`✗`), `⛔` deny-override / record-hold lines (D2-08), the `✗ REJECTED` rogue-credential outcome card, and the WITHHELD state.
- **Amber** (`amber-100`/`amber-900` pill, `amber-200` banner): per-domain pointer/tier labels AND — load-bearing — the **`[MOCK]` tag on the rejected rogue credential** (D2-10) and the persistent `[DEMO / MOCK]` banner (MODEL-03). Amber is the "simulated/untrusted, do not read as a real decision" signal; it is deliberately distinct from green/red.
- **Slate** (`slate-700` solid / `slate-100` pill): the **active stage-trigger button** and the throwaway view-toggle buttons (neutral; carry no decision authority), neutral attributes (unit/domain pointers, compartments), and disabled-on-wrong-turn triggers (`disabled:opacity-40`, the proven disabled treatment from `DecisionExplorer.tsx:242`).
- **Current-stage highlight (D2-05):** the enabled/current stage trigger is the solid slate button (`bg-slate-800 text-white`); not-yet-reached and already-completed triggers are disabled/dimmed (`disabled:opacity-40`). The highlight is by enabled-vs-dimmed state + a slate "current" ring, NOT by introducing a new accent hue.

## Interaction Contract (federation-specific)

Documented here because Phase 2's value is interaction, and the checker/planner/executor need the state-machine contract explicit. All state lives in the single `useReducer` world-state (D2-01); views read via `useWorld()` and dispatch — no second source of truth.

| Interaction | Contract |
|-------------|----------|
| Interim view switch (D2-04) | `activeView: 'explorer' \| 'federation'` as **local `useState` in `DemoRoot`** (NOT world-state, NOT router, NOT `routeTree.gen.ts`). Render a two-button toggle row **between `RoleSwitcherHeader` and `<main>`** so no prop-drilling and the banner/header are never omitted (R7). Throwaway — Phase 4 replaces it. *(Resolves RESEARCH open question #2.)* |
| Four-step stage machine (FED-02, D2-05) | `fedRunStage: IDLE → PUBLISHED → DISCOVERED → REQUESTED → RESPONDED → (New run → IDLE)`. Each trigger enabled ONLY on its turn; current is the solid-slate button, others dimmed (`disabled:opacity-40`). Each trigger appends its typed `Envelope`(s) to `fedTranscript` (DISCOVER appends DISCOVER + DISCOVER_RESULT in one action). |
| Transcript scope (FED-02, D2-06) | The transcript pane shows ONLY the current run's envelopes; "New run" (`FEDERATION_RESET`) clears `fedTranscript` and resets stage to IDLE. REQUEST_DETAIL/DETAIL_RESPONSE ALSO persist into the holder's `fedInbox` and the requester's `fedOutbox` (append-only, survive resets). |
| Holder-gated release (FED-04, D2-07/D2-08) | On REQUEST_DETAIL, the holder unit's ABAC auto-decides (`evaluate` + `releaseRequirementFor`); the inbox surfaces the ALLOW/DENY with the reused `DecisionTrace`. A held/revoked record forces DENY via a `⛔` override line even if the requester clears. **No human override.** |
| Verify-before-trust (FED-03, D2-10) | The holder runs `verifyCredential` BEFORE any ABAC evaluation; a failed verify → DENY "credential not verified", ABAC is never run on unverified claims. The dedicated Credential Verify panel demonstrates both outcomes side by side. |
| Credential auto-verify (FED-03, D2-09) | Credentials are signed async once at mount (`CREDENTIALS_READY`); the Credential Verify panel **auto-verifies on mount** (a `useEffect` keyed on `fedCredentials`) so both outcomes are visible on scroll without a click. An optional "Re-verify" button MAY be offered for interactivity. *(Resolves RESEARCH open question #3.)* |
| Default exchange selection (FED-02) | Default the transcript run to a **guaranteed-ALLOW** subject·requester·holder triple so the FED-04 accept path is one click (e.g. a within-unit / agreement-linked pair). A guaranteed-DENY pair (no-agreement, e.g. MILITARY_1 → INTEL) is reachable via the dropdowns to demonstrate the withheld path. *(Resolves RESEARCH open question #1.)* |
| Unit-console picker (FED-04) | Local `useState` view selection only — does NOT dispatch to the store (display choice, not a decision input). |
