Build the single shared in-memory world-state store (MODEL-02): ONE `useReducer` holding the entire world behind the React.dev split-context pattern, with the 6 core actions and an append-only event log. This is the integration seam every later phase plugs into — get it immutable and event-sourced now. The decision is DERIVED in the view (plan 01-04) via `useMemo`, NEVER stored here.

Purpose: MODEL-02 demands ONE source of truth (no per-spike fragmentation). The store must return new subject objects on mutation or the live-recompute breaks (R2, the locked anti-pattern), and it must event-source every mutation so Phase 3's `reconstructSubject`/`whoCanAccess` plug in unchanged (R6). REQUEST_ATTRIBUTE must mutate nothing — that is the separation-of-duties crux that makes ROLE-02 real.
Output: A Vitest-tested store (`world-state.tsx`) that the Decision Explorer (01-04) consumes via `useWorld()` / `useWorldDispatch()`.

  <behavior>
    - SET_ROLE replaces currentRole; no event is appended; subjects/resources are referentially unchanged.
    - SET_TARGET replaces abacTarget (subjectId + resourceId + domain); no event is appended (it is a view-selection action, not a mutation).
    - APPROVE_ATTRIBUTE adds a compartment to the target subject: the subject object is a NEW reference, its compartments array is a NEW array containing the granted compartment, one AttrEvent (op GRANT_COMPARTMENT, value = the compartment, actor = the current role label) is appended, and seq increments by 1.
    - REVOKE_ATTRIBUTE removes a compartment: NEW subject ref, compartment absent, one AttrEvent (op REVOKE_COMPARTMENT) appended.
    - TOGGLE_SECURITY_HOLD flips the target subject's flags.securityHold: NEW subject ref with a NEW flags object, one AttrEvent (op SET_HOLD when turning on, CLEAR_HOLD when turning off) appended.
    - REQUEST_ATTRIBUTE appends one AttrEvent (op-equivalent request entry, actor = current role label) but performs ZERO attribute mutation — every subject object is referentially identical to before (the SoD crux).
    - The state shape contains no `decision` key and the reducer rejects/omits any RECOMPUTE action — decision is derived in the view, never stored.

    Create `frontend/src/demo/store/world-state.tsx`. Import the world types from `../lib/model` and the seed arrays from `../lib/seed`.

    STATE SHAPE — define `interface WorldState { units; subjects: Subject[]; resources: Resource[]; events: AttrEvent[]; hubIndex: HubPointer[]; currentRole: RoleId; abacTarget: { subjectId: string; resourceId: string; domain: Domain }; seq: number }`. There must be NO `decision` field (R2). `seedWorld(): WorldState` builds the initial state from the seed arrays with `currentRole` defaulting to `"ACCESS_APPROVER"`, `abacTarget` defaulting to the CA-1 clean-ALLOW triple (the seed's first same-unit subject+resource+its domain), `events: []`, `seq: 0`.

    ACTION UNION — define `type Action` with exactly these 6 variants: `{ type: "SET_ROLE"; role: RoleId }`, `{ type: "SET_TARGET"; subjectId?: string; resourceId?: string; domain?: Domain }`, `{ type: "APPROVE_ATTRIBUTE"; subjectId: string; value: Compartment }`, `{ type: "REVOKE_ATTRIBUTE"; subjectId: string; value: Compartment }`, `{ type: "TOGGLE_SECURITY_HOLD"; subjectId: string }`, `{ type: "REQUEST_ATTRIBUTE"; subjectId: string; value: Compartment }`. Do NOT add a RECOMPUTE action (R2).

    REDUCER — `function reducer(state: WorldState, action: Action): WorldState`. SET_ROLE: return `{ ...state, currentRole: action.role }` (no event, subjects untouched). SET_TARGET: return `{ ...state, abacTarget: { ...state.abacTarget, ...defined fields } }` (no event). For the three mutating actions (APPROVE_ATTRIBUTE, REVOKE_ATTRIBUTE, TOGGLE_SECURITY_HOLD): map over `state.subjects`, and for the matched subjectId return a NEW subject object (clone via the spike `cloneSubject` idiom — `{ ...s, compartments: [...s.compartments], flags: { ...s.flags } }`) with the mutation applied (APPROVE adds the compartment if absent; REVOKE filters it out; TOGGLE flips `flags.securityHold`), leaving all other subjects referentially identical; append ONE AttrEvent built by a helper `appendEvent` and increment `seq`. REQUEST_ATTRIBUTE: append ONE AttrEvent ONLY — return the SAME subjects array references (zero mutation, the SoD crux); increment seq. The event `actor` field for every appended event = `ROLES[state.currentRole].label`. Map the action to the AttrEvent `op`: APPROVE→GRANT_COMPARTMENT, REVOKE→REVOKE_COMPARTMENT, TOGGLE on→SET_HOLD / off→CLEAR_HOLD, REQUEST→GRANT_COMPARTMENT-style request entry (use a distinct request marker in `value`/actor copy so the log reads "requested … (cannot grant)" — match the spike copy at Spike004Sod.tsx:131-138). Use `SET_REVOKED`/`CLEAR_REVOKED` and `AUTHORIZE_SUBJECT`/`WITHDRAW_AUTHORIZATION` ops are NOT dispatched in Phase 1 actions but remain valid AttrOp members (D-11/R6 forward-compat) — do not remove them from the type.

    SPLIT-CONTEXT (React.dev pattern, per RESEARCH Wave 2) — create `const WorldStateContext = createContext<WorldState | undefined>(undefined)` and `const WorldDispatchContext = createContext<Dispatch<Action> | undefined>(undefined)`. Export `function WorldProvider({ children }: { children: React.ReactNode })` that calls `const [state, dispatch] = useReducer(reducer, undefined, seedWorld)` (lazy init) and nests `<WorldStateContext.Provider value={state}><WorldDispatchContext.Provider value={dispatch}>{children}</WorldDispatchContext.Provider></WorldStateContext.Provider>`. Export TWO guarded hooks mirroring `auth-context.tsx:90-96`: `useWorld()` returns the state and throws `"useWorld must be used within a WorldProvider"` if the context is undefined; `useWorldDispatch()` returns the dispatch and throws `"useWorldDispatch must be used within a WorldProvider"` if undefined. Optionally export a tiny selector helper `useCurrentRole()`/`useAbacTarget()` reading from `useWorld()`, but the two core hooks are required.

    Use `useReducer` ONLY — NO Redux/Zustand/Jotai or any new state library (MODEL-02 / PROJECT.md constraint). This file is `.tsx` because it returns JSX from the provider.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| viewer action → store mutation | A viewer (acting as a role) dispatches actions; the reducer is the only mutation path. All data is in-memory/seeded. |
| operating role → action authority | The store records `actor` on every event; whether a role MAY dispatch a given action is enforced in the view (01-04) by op membership — the store records who acted. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering (stale/second source of truth) | reducer immutability | mitigate | Reducer returns NEW subject objects on mutation (cloneSubject idiom); a test asserts new-ref + flipped decision, so a stale-mutation regression fails CI (R2). |
| T-03-02 | Repudiation | append-only event log | mitigate | Every mutating action appends one AttrEvent with `actor` = role label and an increasing `seq`; tested. Superset of the spike shape so Phase-3 replay reconstructs "who did what as of T" (R6). |
| T-03-03 | Elevation of privilege (SoD bypass) | REQUEST_ATTRIBUTE | mitigate | REQUEST_ATTRIBUTE performs zero attribute mutation by construction; a test asserts subject references are unchanged. The Manager cannot grant via the store. (View-level op-gating is added in 01-04.) |
| T-03-04 | Spoofing (mock-as-real authorization state) | in-memory world-state | mitigate | The store holds only seeded MOCK data; the [DEMO / MOCK] banner + [MOCK] tags (01-01 / 01-04) ensure the viewer never reads store state as live authorization. No production trust signal exists here. |
| T-03-SC | Tampering | npm installs | accept | No new packages installed (React useReducer/createContext + Vitest already vendored); package legitimacy gate not triggered. |

No unmitigated HIGH-severity threat. Threats are proportional to an in-memory mock store (stale-state correctness and SoD legibility), not production authorization enforcement.
