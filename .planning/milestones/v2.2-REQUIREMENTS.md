## Digital Resource Model (RSRC)

- [ ] **RSRC-01**: Digital resources are organized in a 3-tier hierarchy: **Network → Platform → Application**.
- [ ] **RSRC-02**: Each resource carries a classification level drawn from the 5-tier clearance ladder (`UNCLASSIFIED → RESTRICTED → CONFIDENTIAL → SECRET → TOP_SECRET`).
- [ ] **RSRC-03**: Network classification examples (indicative, not exhaustive): National Restricted (`RESTRICTED`), National Secret (`SECRET`), Tactical Secure (`SECRET`), NATO Restricted (`RESTRICTED`), NATO Secret (`SECRET`), NATO Top Secret (`TOP_SECRET`).
- [ ] **RSRC-04**: Each resource carries `admin_org_id` (controls/delegates access) and `asset_owner_org_id` (owns the resource) — mirrors v2.1 zone model.
- [ ] **RSRC-05**: A Platform belongs to exactly one Network. An Application belongs to exactly one Platform. (Strict tree; no multi-homing in demo scope.)

---

## Access Rules (RSRC-ACCESS)

- [ ] **RSRC-ACCESS-01**: Access to a Network requires clearance ≥ network classification level AND explicit authorization for that specific network.
- [ ] **RSRC-ACCESS-02**: Access to a Platform requires an active Network grant for the platform's host network AND explicit authorization for the platform.
- [ ] **RSRC-ACCESS-03**: Access to an Application requires an active Platform grant for the application's host platform AND explicit authorization for the application.
- [ ] **RSRC-ACCESS-04**: Physical zone access may be a prerequisite for platform/network access — a zone grant for the room/building where the terminal resides can be required (link to v2.1 zone model).
- [ ] **RSRC-ACCESS-05**: Access resolution for a resource applies gate chain: (1) clearance check, (2) explicit grant per tier, (3) prerequisite tier grants active.

---

## Access Grants (RSRC-GRANT)

- [ ] **RSRC-GRANT-01**: A `ResourceAccessGrant` links a person to a specific resource (Network, Platform, or Application) with `valid_from` and `valid_until` (nullable = permanent).
- [ ] **RSRC-GRANT-02**: Grants are per-resource; each tier always requires explicit authorization — no automatic inheritance across tiers (Network grant does not grant Platform access).
- [ ] **RSRC-GRANT-03**: Time-limited grants reflect real-world temporary access (e.g., external contractor on National Restricted for 30 days).

---

## Delegation (RSRC-DELEG)

- [ ] **RSRC-DELEG-01**: `admin_org` can delegate resource access-granting authority for a resource to a named person OR another org — mirrors v2.1 `ZoneAccessDelegate`.

---

## Mock Dataset (RSRC-SEED)

- [ ] **RSRC-SEED-01**: Dataset defines ≥3 Networks with realistic classification tiers from the 6-unit scenario.
- [ ] **RSRC-SEED-02**: Dataset includes Platforms (terminals, servers, workstations) on each network.
- [ ] **RSRC-SEED-03**: Dataset includes Applications (systems, services) on each platform.
- [ ] **RSRC-SEED-04**: Dataset demonstrates the zone-prerequisite link (access to a network terminal requires zone access to the room it's in).
- [ ] **RSRC-SEED-05**: Grants include active, expired, and future-dated examples across all three resource tiers.

## Demo UI (RSRC-UI)

- [ ] **RSRC-UI-01**: Demo includes a Resource Browser: Network → Platform → Application hierarchy with classification badges.
- [ ] **RSRC-UI-02**: Selecting a resource shows `admin_org`, `asset_owner_org`, classification, active grants, delegates.
- [ ] **RSRC-UI-03**: Access Resolution Explorer for digital resources: select person + resource, compute ALLOW/DENY with gate chain trace (clearance, tier grants, zone prerequisite if applicable).
